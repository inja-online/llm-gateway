**Decision:** **error** on the **translation** path (not silent skip)

Agents often register tools that are only meaningful on one vendor. The gateway treats that carefully so clients never believe a tool was registered when it was dropped.

## Policy

| Path | Non-function `tools[].type` (e.g. `custom`, `file_search`, server tools) |
|---|---|
| **Passthrough** (OpenAI → `openai` / `openai_compat`) | **Forward as-is** (JSON map; model rewrite only). Upstream may accept or reject. |
| **Translation** (OpenAI → Anthropic/Google via canonical IR) | **HTTP 400** `invalid_request_error` — only `type: "function"` (or empty type, treated as function) is supported. |

**Rationale:** silent skip made agents believe tools were registered when they were not. Fail closed on translate so clients fix the tool set or use passthrough to a host that supports custom tools.

## Function tools

Supported on both passthrough and translate:

- **Name** is required  
- **Parameters** JSON Schema is carried on the canonical IR  
- Each dialect egress rebuilds the vendor-shaped tool definition  

Tool results and tool-use round-trips follow the same family rules as chat (passthrough vs IR).

## Practical guidance

1. Prefer **passthrough** when you need vendor-specific tool types.  
2. On **cross-family** routes, send only function tools (or drop tools client-side).  
3. Watch for future expansions: [#107](https://github.com/inja-online/llm-gateway/issues/107) (tool union), [#161](https://github.com/inja-online/llm-gateway/issues/161) (custom grammar tools).
